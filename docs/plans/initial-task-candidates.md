# 初期タスク候補

## 概要

この文書は、現在の技術スタック方針案に沿って、研究環境と Web プロトタイプのそれぞれで最初に何を作るかを整理するためのメモである。

本プロジェクトでは、研究・解析とプロトタイプ制作の役割を分けて考える。そのため、初期タスクも単一の一本道として決めるのではなく、それぞれの目的に応じた候補をいくつか持ちながら比較検討する。

## 基本的な考え方

研究環境では、理解、検証、比較、観察を優先する。  
Web プロトタイプでは、触れること、音が変わること、構造と結果の関係が見えることを優先する。

つまり、研究環境は「モデルを理解するための場」、Web プロトタイプは「モデルを体験しながら試すための場」として考える。

## 1. 研究環境で最初に作る候補

### 候補 1: 最小の音声解析ノートブック

入力音声を読み込み、波形、スペクトル、スペクトログラム、基本周波数、フォルマントなどを観察できるノートブックを作る。

#### 目的

- 比較基盤を作る
- 音声解析の作業環境を整える
- 今後生成する音の観察手段を先に持つ

#### 強み

- 最も堅実で、後から無駄になりにくい
- 研究環境の土台になる
- 自作モデルだけでなく参考音声の分析にも使える

#### 弱み

- 単体では音声生成そのものは進まない
- 見た目の進捗としては地味

### 候補 2: 単純な source-filter 実験環境

簡易的な声帯励振とフォルマントフィルタを組み合わせ、母音らしい音を出す実験環境を作る。

#### 目的

- 分析と合成を最短距離でつなぐ
- 最小限の音声生成実験を始める

#### 強み

- 音が出るまでが早い
- 解析手法との往復がしやすい
- 母音生成の最初の比較対象になる

#### 弱み

- 発声器官シミュレーションとしてはまだ抽象度が高い
- 物理モデルとしての納得感は弱い

### 候補 3: 声帯モデル単体の実験環境

2-mass 系や簡易 glottal source を実装し、周期、開閉、息漏れ成分などの変化を観察する。

#### 目的

- 励振源を独立に理解する
- 声帯側のパラメータ設計を早い段階で掴む

#### 強み

- 発声の源の部分を切り出して学べる
- 後で声道モデルと接続しやすい

#### 弱み

- 単体では最終音声に直結しにくい
- 初期段階ではやや局所的なテーマになりやすい

### 候補 4: 声道共鳴モデル単体の実験環境

チューブ長や断面積列を変えながら、共鳴やスペクトルの変化を見る実験環境を作る。

#### 目的

- 声道側の寄与を独立に理解する
- 軽量な伝搬モデルの性質を掴む

#### 強み

- 発声器官シミュレーションの本筋に近い
- 形状と音の関係を直接見やすい

#### 弱み

- 励振源を簡略化しないと実験しにくい
- 実装内容によっては少し数理寄りになる

### 候補 5: 母音ターゲット比較用の実験環境

既知の母音サンプルと、自作モデルの出力を並べて比較する環境を作る。

#### 目的

- 評価の軸を作る
- 目標音との距離感を把握する

#### 強み

- 研究として意味がある
- 感覚ではなく比較に基づいて調整できる

#### 弱み

- 単体では生成モデルの実装が前提になる
- 比較基準の整備が必要

### 候補 6: 文献再現ノートブック

文献に出てくる簡易モデルや式を、小規模なノートブックとして再現する。

#### 目的

- 文献理解を実装に落とす
- 仮説ではなく既存知見を確認する

#### 強み

- 足場が堅い
- 調査結果と実装を結びつけやすい

#### 弱み

- 成果が断片的になりやすい
- そのままプロトタイプに直結しないこともある

## 2. Web プロトタイプで最初に作る候補

### 候補 1: 母音を出せる最小インタラクティブ UI

数個のスライダーで声帯や声道の主要パラメータを変え、即座に音が変わるプロトタイプを作る。

#### 目的

- 最小の触れる体験を成立させる
- 音が出ることと制御できることを最優先で確認する

#### 強み

- 最初の到達点として分かりやすい
- 実装範囲が比較的限定される
- 今後の UI 実験の基礎になる

#### 弱み

- 可視化が弱いと方向性が伝わりにくい
- 体験としてはやや素朴になりやすい

### 候補 2: Pink Trombone 風の 2D 操作プロトタイプ

声道形状を簡略な 2D で表示し、マウス操作などで変形できるプロトタイプを作る。

#### 目的

- プロジェクトの方向性を早く確認する
- 形状変化と音の変化を直感的に結びつける

#### 強み

- このプロジェクトらしさが出やすい
- 視覚と聴覚の連動が分かりやすい
- 今後の 2D / 3D 方針の判断材料になる

#### 弱み

- UI 設計の難易度がやや上がる
- 初期段階では見た目に工数を吸われる可能性がある

### 候補 3: 音と解析を同時に見るデバッグ UI

音を出しながら、波形、スペクトル、簡易フォルマントなどを同一画面で観察できるプロトタイプを作る。

#### 目的

- 研究環境と Web 試作を橋渡しする
- 開発中の観察と調整をしやすくする

#### 強み

- デバッグ用途として長く使える
- 音だけでなく内部状態も追いやすい

#### 弱み

- 見せるプロトタイプとしては地味になりやすい
- 初期 UI が実験ツール寄りになる

### 候補 4: 声帯と声道を分けて操作する UI

ピッチや開閉、息成分と、声道形状や constriction を別パネルで操作できるようにする。

#### 目的

- モデル構造の分離を UI にも反映する
- 調整対象を整理しやすくする

#### 強み

- 設計思想が明確になる
- 将来的な拡張に耐えやすい

#### 弱み

- 初期段階では少し説明的で複雑になりやすい

### 候補 5: 母音プリセット付きプロトタイプ

`/a/`, `/i/`, `/u/` などのプリセットを用意し、そこから手動で編集できるプロトタイプを作る。

#### 目的

- 最初の体験の分かりやすさを上げる
- 調整前後の比較をしやすくする

#### 強み

- 触った人がすぐ意味を理解しやすい
- 学習コストが低い

#### 弱み

- プリセット設計の手間が追加で発生する

### 候補 6: 2D 可視化と録音再生を持つプロトタイプ

生成音を録音し、後から再生・比較できるプロトタイプを作る。

#### 目的

- 試行錯誤の結果を残しやすくする
- 比較と評価をしやすくする

#### 強み

- 実験の履歴を扱いやすい
- 研究用途にも寄せやすい

#### 弱み

- 初期段階としては少し範囲が広い

## 3. 組み合わせ案

初期タスクは、研究環境と Web プロトタイプを組み合わせて考えると見通しが良い。

### 案 A: 安全な案

#### 研究環境

- 最小の音声解析ノートブック

#### Web プロトタイプ

- 母音を出せる最小インタラクティブ UI

#### 特徴

もっとも堅実で、初期の基盤作りに向いている。派手さは少ないが無駄になりにくい。

### 案 B: バランス案

#### 研究環境

- 最小の音声解析ノートブック
- 声道共鳴モデル単体の実験環境

#### Web プロトタイプ

- 音と解析を同時に見るデバッグ UI

#### 特徴

観察、理解、試作のバランスが良い。研究とプロトタイプの接続も意識しやすい。

### 案 C: 方向性重視案

#### 研究環境

- 文献再現ノートブック
- 母音ターゲット比較用の実験環境

#### Web プロトタイプ

- Pink Trombone 風の 2D 操作プロトタイプ

#### 特徴

このプロジェクトらしさが早く見えるが、初期負荷はやや高い。

## 4. 現時点で有力な出発点

現時点では、以下の出発点がもっとも自然に見える。

### 研究環境

- 最小の音声解析ノートブック
- 最小の合成実験、または声道共鳴モデル単体の実験環境

### Web プロトタイプ

- 母音を出せる最小インタラクティブ UI
- 必要に応じて簡易可視化または解析表示を追加

この構成であれば、研究側で観察基盤を整えつつ、Web 側で「触れる最小体験」を早めに成立させることができる。

## 5. 次に決めるべきこと

この文書の次の段階では、以下を決める必要がある。

1. 研究環境の第一歩を解析重視にするか、合成重視にするか
2. Web プロトタイプの第一歩を UI 重視にするか、可視化重視にするか
3. 研究環境と Web プロトタイプの間で、どのデータやパラメータを共有するか
4. 最初の到達点をどの程度の小ささに抑えるか
