# 技術スタック方針案

## 概要

この文書は、`voice-simulator` における技術スタックの大まかな方針案を整理したものである。現時点では、最終的なプロダクトのための技術スタックを急いで確定させるのではなく、まず研究・解析とプロトタイプ制作に適した技術スタックを分けて考えることを前提とする。

本プロジェクトは、研究とプロトタイプ制作の試行錯誤を経たのちに最終的なプロダクトへ進む構想である。そのため、最初から単一の技術スタックに全てを統一するよりも、目的ごとに適した環境を選ぶ方が合理的である。

## 基本方針

現時点では、技術スタックを大きく以下の三層に分けて考える。

1. 研究・解析のための技術スタック
2. プロトタイプ制作のための技術スタック
3. 最終的なプロダクトのための技術スタック

このうち、今の段階で具体的に方針を定めるのは 1 と 2 であり、3 は候補を把握するに留める。

## 1. 研究・解析のための技術スタック

### 第一候補

- Python
- NumPy
- SciPy
- Jupyter

### 必要に応じて追加する候補

- matplotlib
- librosa
- PyVista
- Taichi
- TorchAudio

### この構成を有力とする理由

研究・解析の段階では、文献を読みながら数式やモデルを試し、FFT、スペクトル、フォルマント、LPC、ケプストラムなどの解析を素早く行えることが重要である。その点で、Python を中心とした科学計算環境は最も扱いやすい。

また、Jupyter を用いれば、調査メモ、実験コード、可視化、観察結果を一体で残しやすい。これは、プロジェクト初期の探索において大きな利点となる。

### この段階で期待する役割

- 文献に出てくるモデルの小規模な再現
- 音声解析や信号処理の試行
- 実験用スクリプトの作成
- 比較用データの読み込みと分析
- モデルの挙動観察と可視化

### 補足候補

Julia も数値計算や信号処理の面では魅力があるが、現時点では主軸候補というより補助的な実験環境として考える方が自然である。

## 2. プロトタイプ制作のための技術スタック

### 第一候補

- TypeScript
- Web Audio API
- AudioWorklet
- React または Svelte
- Canvas または Three.js

### この構成を有力とする理由

プロトタイプ制作の段階では、高忠実度よりも、まず触れること、音の変化をすぐ確認できること、構造や挙動を視覚的に示せることが重要になる。その点で、Web 技術は非常に相性がよい。

特に、Pink Trombone に近い方向性の「インタラクティブに触って理解できるシミュレータ」を作るには、ブラウザベースの UI と可視化は有力である。共有しやすく、試してもらいやすい点も強みである。

### この段階で期待する役割

- リアルタイムに触れるシミュレータの試作
- 声帯や声道パラメータのインタラクティブ操作
- 2D または簡易 3D の可視化
- 実験用 UI の構築
- 音の変化を素早く確認する環境の整備

### 第二候補

- Rust
- CPAL
- rodio
- egui
- wgpu
- nannou

Rust 中心のネイティブ試作は、Web より立ち上がりは重いが、リアルタイム音声処理と将来的な実装基盤の連続性を見たい場合には有力である。初期段階では第一候補ではないが、将来を見据えた試作環境として十分検討価値がある。

## 3. 最終的なプロダクトのための技術スタック

現時点では、最終的なプロダクトのための技術スタックはまだ確定しない。ここは、研究とプロトタイプ制作を進めた結果を踏まえて判断する。

### 現時点での主な候補

- C++ / JUCE
- Rust
- Web + ネイティブコアのハイブリッド構成
- Qt + C++
- Unity + ネイティブ DSP

### 現時点で確定しない理由

最終プロダクトに求める要件は、今後の研究とプロトタイプの結果によってかなり変わる可能性がある。たとえば、どの程度のリアルタイム性が必要か、どこまで 3D 可視化を重視するか、スタンドアロンアプリにするか、Web 展開も重視するかによって、最適な選択は変わる。

そのため、今は無理に決めず、候補を意識しながら材料を集める段階とする。

## 現時点での大まかな結論

現段階では、以下の構成がもっとも自然に見える。

### 研究・解析

- Python を主軸とする

### プロトタイプ制作

- TypeScript + Web Audio API を第一候補とする

### 最終プロダクト

- Rust または C++ / JUCE を有力候補として保留する

## この方針の利点

この方針には、以下の利点がある。

- 研究の速さと試作の速さを両立しやすい
- 初期段階で重い実装判断を避けられる
- プロトタイプで得た知見をもとに、最終プロダクトの要件を整理しやすい
- 研究環境と実行環境の責務を分けられる

## 今後の検討課題

今後は、以下を見ながら方針を更新していく。

1. Python 環境でどの程度まで研究用モデルを回せるか
2. Web 技術でどの程度までリアルタイム試作が可能か
3. Rust をプロトタイプ段階から導入する価値があるか
4. 2D / 3D 可視化をどの段階でどこまで扱うか
5. 研究コードとプロトタイプ実装をどう接続するか

## 備考

この文書は、プロジェクト初期段階における技術スタックの方針案であり、今後の実装や比較検討の結果に応じて更新される前提の草稿である。
