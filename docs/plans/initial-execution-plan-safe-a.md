# 初期実行計画: 案 A

## 概要

この文書は、[initial-task-candidates.md](/Users/dolphilia/github/voice-simulator/docs/plans/initial-task-candidates.md) で整理した `案 A: 安全な案` を、より小さい実行単位に分解した計画である。

`案 A` は、初期段階で無理に広げず、研究環境と Web プロトタイプの両方において、後から無駄になりにくい基盤を先に作る方針である。

## この計画の対象

### 研究環境

- 最小の音声解析ノートブック

### Web プロトタイプ

- 母音を出せる最小インタラクティブ UI

## 全体方針

この段階では、次のことを優先する。

1. 小さく終わる単位で進めること
2. それぞれ単独でも意味がある成果にすること
3. 研究環境と Web 側を緩く接続し、後で比較や共有がしやすい形にすること

## 研究環境の実行計画

研究環境では、最初から高度な発声モデルには進まず、まず音を観察し、比較するための最小基盤を作る。

### ステップ 1: 研究環境の最小セットアップ

#### 目的

Python ベースで実験と解析を進められる最小環境を整える。

#### 作るもの

- 実験用ディレクトリ構成
- Python 実行環境の方針メモ
- 依存関係の最小候補一覧

#### 完了条件

- どこにノートブックを置くかが決まっている
- 最低限使うライブラリ候補が明確になっている

### ステップ 2: 最小の音声読み込み

#### 目的

参照用の音声データを読み込み、後続の解析で扱える状態にする。

#### 作るもの

- WAV などの音声を読み込むノートブック
- サンプリング周波数、長さ、振幅範囲を確認する処理

#### 完了条件

- 任意の入力音声を読み込める
- 音声の基本情報を表示できる

### ステップ 3: 基本的な可視化

#### 目的

音声を耳だけでなく視覚的にも観察できるようにする。

#### 作るもの

- 波形表示
- スペクトル表示
- スペクトログラム表示

#### 完了条件

- 一つの入力音声に対して、基本的な可視化が一通りできる

### ステップ 4: 基本指標の観察

#### 目的

音の比較に使う最小限の解析指標を持つ。

#### 作るもの

- 基本周波数の観察
- フォルマント推定の試行
- 必要に応じて簡単な補助指標

#### 完了条件

- 少なくとも一つの音声サンプルに対して、基本周波数とフォルマントの概形を確認できる
- 解析結果が妥当かどうかを目視で点検できる

### ステップ 5: 比較しやすいノートブックに整理

#### 目的

今後、自作音声と参考音声を比較できる形に整える。

#### 作るもの

- 入力ファイルを差し替えて再利用できるノートブック
- 観察項目を一定の順序で並べたテンプレート

#### 完了条件

- 一つのノートブックを複数の音声に再利用できる
- 今後の比較実験の土台として使える

## Web プロトタイプの実行計画

Web 側では、まず「最小でも触って音が変わる」状態を作る。見た目の凝り込みよりも、操作と音の対応を成立させることを優先する。

### ステップ 1: Web 開発環境の最小セットアップ

#### 目的

TypeScript と Web Audio API で試作できる最低限の土台を作る。

#### 作るもの

- Web プロトタイプ用ディレクトリ構成
- 最小の開発サーバー環境
- TypeScript ベースの雛形

#### 完了条件

- ブラウザで空のプロトタイプ画面を開ける
- 音声処理コードと UI コードを分けて置ける構造になっている

### ステップ 2: 最小の音出し

#### 目的

ブラウザ上で安定して音を出せることを確認する。

#### 作るもの

- Web Audio API または AudioWorklet を用いた最小の発音処理
- 音を開始・停止するための最小操作

#### 完了条件

- ボタン操作で音を開始・停止できる
- 音切れやエラーなく最小の連続音を出せる

### ステップ 3: 最小の母音らしい音

#### 目的

ただの発振音ではなく、母音に近い方向の音を最小構成で作る。

#### 作るもの

- 簡易的な励振源
- 最小限の共鳴・フィルタ処理

#### 完了条件

- 単純な連続音よりも、ある程度母音らしい音色変化を感じられる

### ステップ 4: 最小の操作 UI

#### 目的

音を変えられることを、画面上の操作として成立させる。

#### 作るもの

- 主要パラメータを変更するスライダー
- 現在値の表示
- 音との結びつきが分かる最小 UI

#### 完了条件

- 少なくとも 2 から 4 個程度のパラメータを動かし、音の変化を確認できる

### ステップ 5: 最小の視覚的補助

#### 目的

体験の理解を助けるために、簡易な可視要素を加える。

#### 作るもの

- パラメータ状態の表示
- 簡単な波形、レベル表示、または模式的な声道状態表示

#### 完了条件

- 音だけでなく、現在の状態を画面上で最低限追える

## 研究環境と Web 側の接続

この段階では、両者を強く統合する必要はない。ただし、後で比較しやすいように最低限の接点は意識する。

### 接続候補

- パラメータ名の考え方をできるだけ揃える
- サンプリング周波数や基本的な信号の扱いを意識して揃える
- Web 側で出した音を研究側で解析できるようにする

### この段階でまだやらないこと

- 共通コアの実装
- 高度なモデル共有
- 自動評価基盤の整備

## 小さい到達点

この計画における最初の到達点は、次の二つである。

### 到達点 1

研究環境で、参照音声を読み込み、波形、スペクトル、スペクトログラム、基本周波数、フォルマントを観察できる。

### 到達点 2

Web 上で、最小の UI を通じて母音らしい連続音を出し、いくつかのパラメータ変更による音の違いを確認できる。

## この段階で保留するもの

以下は次のフェーズに回す。

- 本格的な声道形状可視化
- Pink Trombone 風の 2D 直接操作
- 子音や乱流の扱い
- 高忠実度の声帯モデル
- 研究環境と Web プロトタイプの深い統合

## 実行順の提案

現時点では、次の順序が自然に見える。

1. 研究環境のステップ 1 から 3 を先に進める
2. 並行して Web 側のステップ 1 と 2 を進める
3. 研究環境で解析の見え方を固める
4. Web 側で最小の母音らしい音と UI を成立させる
5. 最後に、Web で出した音を研究環境で観察する流れを作る

## 備考

この文書は、初期フェーズの実行を小さく始めるための草稿であり、実際に手を動かしながら必要に応じて更新する前提の計画である。
