# 技術スタック候補比較メモ

## 概要

この文書は、`voice-simulator` の技術スタックを検討するための候補整理メモである。現時点では、急いで一つに決めることは目的ではない。むしろ、プロジェクトの目的に対してどのような選択肢があり、それぞれが何に強く、何に弱いのかを広く把握することを目的とする。

本プロジェクトが技術スタックに求める条件は、大きく次の四つである。

1. 音声処理やリアルタイム音声入出力を扱いやすいこと
2. 数値計算や物理シミュレーションを進めやすいこと
3. 可視化やインタラクティブな操作系を構築しやすいこと
4. 将来的に高度化しても破綻しにくいこと

これを前提に、候補をいくつかの系統に分けて整理する。

## 比較の視点

技術スタックを評価する際には、少なくとも以下の観点が重要になる。

- リアルタイム音声処理のしやすさ
- 数値計算、物理シミュレーションとの相性
- 2D / 3D 可視化のしやすさ
- GUI やインタラクションの作りやすさ
- 試作の速さ
- 性能と制御性
- 将来の保守性
- 周辺ライブラリやコミュニティの厚さ

## 候補 1: Web 中心のスタック

### 例

- TypeScript / JavaScript
- [Web Audio API](https://developer.mozilla.org/en-US/docs/Web/API/Web_Audio_API)
- AudioWorklet
- [Three.js](https://threejs.org/docs/index.html)
- [React](https://react.dev/) または [Svelte](https://svelte.dev/)
- 必要に応じて [Tauri](https://tauri.app/) でデスクトップ化

### 向いていること

- ブラウザですぐ触れる試作
- インタラクティブな UI
- 2D / 3D の視覚化
- Pink Trombone 的な体験に近いプロトタイプ

### 強み

- 配布しやすく、試してもらいやすい
- UI と可視化が非常に作りやすい
- 音を聞きながら構造を見る体験を作りやすい
- AudioWorklet を使えばブラウザでも比較的きちんとした音声処理ができる

### 弱み

- 重い数値計算や高度な物理シミュレーションには不利
- ブラウザの制約を受ける
- 厳密な低遅延・高性能 DSP はネイティブより不利
- 長期的にコア DSP まで全部 Web に載せるべきかは慎重に見極める必要がある

### 本プロジェクトとの相性

初期フェーズの「触れるシミュレータ」「可視化と操作の実験」には非常に相性が良い。一方で、将来的に高忠実度な数値計算まで全部 Web で押し通すのは重い可能性が高い。

## 候補 2: C++ 中心のネイティブ音響スタック

### 例

- C++
- [JUCE](https://juce.com/juce/)
- 必要に応じて独自 DSP コア
- 追加で OpenGL / Vulkan / Metal 系の描画、または他 UI 技術との併用

### 向いていること

- リアルタイム DSP
- デスクトップ向けの安定した音声アプリ
- 低遅延性を重視する構成
- 将来的なプラグイン化や音響ツール化

### 強み

- 音響ソフトウェアの王道の一つであり、実績が厚い
- 音声入出力、GUI、クロスプラットフォーム対応を比較的一体で扱いやすい
- 性能と制御性が高い
- 将来的に音源やプラグインとして展開する場合にも相性が良い

### 弱み

- 試作速度は Web や Python より遅くなりやすい
- 数値実験や研究ノート的な試行錯誤はやや重い
- 3D 可視化や研究用解析環境は別途工夫が要る

### 本プロジェクトとの相性

最終的に「本格的な音声シミュレーションアプリ」へ寄せる場合は非常に有力である。ただし、初期の探索段階から全面採用すると、検討速度が落ちる可能性がある。

## 候補 3: Rust 中心のネイティブスタック

### 例

- Rust
- [CPAL](https://docs.rs/crate/cpal/latest)
- [rodio](https://docs.rs/rodio/latest/rodio/)
- [wgpu](https://docs.rs/wgpu/)
- [egui](https://www.egui.rs/) または Bevy 系
- [nannou](https://nannou.cc/)

### 向いていること

- 比較的安全性の高いネイティブ実装
- 音声処理と可視化の両立
- 中長期的な保守性を意識した設計

### 強み

- 性能と安全性のバランスがよい
- オーディオ、GPU、GUI の各分野に実用的なライブラリが増えている
- モジュール分離しやすく、長期的な設計に向く
- WebAssembly を含む展開の余地がある

### 弱み

- 音響アプリの定番としては C++ より実績が浅い
- ライブラリ選定の自由度が高いぶん、組み合わせの設計責任が大きい
- 開発体験は良いが、初期の立ち上がり学習コストはある

### 本プロジェクトとの相性

かなり有力な候補である。特に「リアルタイム音声処理」と「将来の堅い実装」の両方を見たい場合に魅力がある。ただし、現時点では C++/JUCE ほど定番の一本道はない。

## 候補 4: Python 中心の研究・試作スタック

### 例

- Python
- [NumPy](https://numpy.org/doc/stable/)
- [SciPy](https://docs.scipy.org/doc/scipy/reference/)
- [Taichi](https://docs.taichi-lang.org/)
- [PyVista](https://docs.pyvista.org/)
- [Jupyter Widgets](https://jupyter.org/widgets)
- 必要に応じて [TorchAudio](https://docs.pytorch.org/audio/stable/index)

### 向いていること

- 文献を読みながらの数値実験
- 音声解析
- 物理モデルの小規模試作
- 可視化を伴う研究用途

### 強み

- 試作が非常に速い
- 数値計算、FFT、解析、実験コードが書きやすい
- Jupyter ベースで調査と実験を一体化しやすい
- Taichi や PyVista を組み合わせると、シミュレーションや 3D 可視化もある程度進められる

### 弱み

- 本格的なリアルタイム音声アプリの中核にはなりにくい
- GUI や配布形態は弱くなりがち
- 長期的に製品コアへそのまま育てるには限界がある

### 本プロジェクトとの相性

研究、比較、解析、モデル実験には非常に向いている。特に初期フェーズでは強力だが、最終的な実行系の本体を Python 単独で担う構図はやや厳しい。

## 候補 5: Julia 中心の数値計算スタック

### 例

- Julia
- [DSP.jl](https://docs.juliadsp.org/dev/)
- SciML 系ライブラリ
- Julia 可視化エコシステム

### 向いていること

- 数値計算と信号処理の実験
- Python より高速寄りの研究コード
- 数理モデルの探索

### 強み

- 数式に近い感覚で書けて、性能も比較的出しやすい
- DSP や数値解析との相性がよい
- 研究用コードとしてはかなり魅力がある

### 弱み

- 音響アプリ開発や GUI の選択肢は主流ではない
- 周辺の実装事例が少ない
- 共同開発や将来の展開まで考えると採用の判断が重くなる

### 本プロジェクトとの相性

研究用途には魅力があるが、現時点では本プロジェクトの主軸に据えるより、補助的な実験環境として考える方が自然である。

## 候補 6: Unity / C# 中心のインタラクティブ 3D スタック

### 例

- C#
- [Unity](https://unity.com/)
- 必要に応じて [Unity Native Audio Plugin SDK](https://docs.unity3d.com/cn/2023.2/Manual/AudioMixerNativeAudioPlugin.html)
- ネイティブ側に C++ DSP

### 向いていること

- 3D 可視化
- 直感的な操作系
- 実験用インターフェースや展示的な体験

### 強み

- 3D 表現、UI、操作性に強い
- シミュレーション結果を見せる体験作りがしやすい
- 必要ならネイティブ DSP を組み合わせられる

### 弱み

- 音声生成コアを Unity 側に寄せすぎると設計が複雑になる
- 音響ソフトウェア本体としては遠回りになる場合がある
- 研究コードや数値実験環境との往復はやや不便

### 本プロジェクトとの相性

3D 表現を強く重視するなら検討価値がある。ただし、発声シミュレーションの本体ではなく、可視化フロントエンドとしての採用を考える方が現実的である。

## 候補 7: openFrameworks / creative coding 系 C++ スタック

### 例

- C++
- [openFrameworks](https://openframeworks.cc/documentation/)

### 向いていること

- 研究と作品制作の中間
- 音と映像を一体にした試作
- 柔軟な可視化

### 強み

- 音声入出力と描画を比較的気軽に扱える
- 試作と表現を両立しやすい
- インタラクティブな実験アプリに向く

### 弱み

- 大規模アプリや堅いアーキテクチャには工夫が必要
- 音響ソフトウェア開発の定番としては JUCE に劣る
- GUI やアプリ構造は自前設計の比重が高くなりやすい

### 本プロジェクトとの相性

探索段階の「触れる研究ツール」としては面白い。ただし、長期的な中核基盤にするかは慎重に見たい。

## 候補 8: Qt 中心の C++ デスクトップスタック

### 例

- C++
- [Qt Multimedia](https://doc.qt.io/qt-6/audiooverview.html)
- Qt Quick / QML

### 向いていること

- デスクトップ向け GUI
- 音声入出力を伴う研究ツール
- 解析・可視化ツール

### 強み

- GUI の完成度を高めやすい
- デスクトップアプリとしてのまとまりを作りやすい
- 音声入出力や可視化を比較的一体で扱える

### 弱み

- JUCE ほど音響特化ではない
- 低遅延 DSP の中核としては工夫が必要
- プラグインや音源展開を強く意識するなら別解の方が自然

### 本プロジェクトとの相性

研究ツールや解析・可視化アプリには向いているが、音声生成コアの主軸としては他候補と比較が必要である。

## 単一スタックで考えない案

このプロジェクトでは、最終的に単一言語・単一環境に全てを寄せない方が合理的である可能性が高い。

有力な分業パターンとしては、次のようなものがある。

### パターン A

- 研究・解析: Python
- 実行コア: Rust または C++
- UI / 可視化: Web

### パターン B

- 研究・解析: Python または Julia
- 実行コアとデスクトップ UI: C++ / JUCE

### パターン C

- 実行コア: Rust
- ネイティブ GUI: egui / nannou / wgpu 系
- 必要に応じて Web フロントエンドも追加

このように分けると、それぞれの得意分野を活かしやすい。

## 現時点で特に有力に見える候補

まだ決定ではないが、現時点で本プロジェクトの目的に照らして特に有力に見えるのは次の系統である。

### 1. Web + ネイティブコアのハイブリッド

最初は Web で可視化と操作系を作り、重い処理やコア DSP を後から Rust または C++ に寄せる構成。初期探索にも将来拡張にも対応しやすい。

### 2. Rust 中心

研究段階の柔軟性と、将来の実装基盤としての堅さの中間を狙える。音声、可視化、保守性をまとめて見たい場合に有力。

### 3. C++ / JUCE 中心

音響ソフトウェアとしての王道を取る構成。最終形を重く見据えるなら強いが、初期探索では少し重い可能性がある。

### 4. Python 併用前提

主軸ではなくても、解析、比較、実験、文献再現のための補助環境として採用価値が高い。

## 現時点でまだ決めない方がよいこと

以下は、もう少し比較してから決める方がよい。

- ブラウザ中心で進めるか、最初からネイティブ中心で進めるか
- コア DSP を最初から低レベル実装するか
- 2D / 3D 可視化を本体とどこまで密結合させるか
- 解析環境と実行環境を分けるかどうか

## 次にやるとよいこと

次の段階では、候補をさらに絞るために、以下を比較するとよい。

2. リアルタイム音声処理のやりやすさ
3. 可視化と操作系の試作速度

5. 将来 3D や高忠実度検証へ進む際の拡張性
